# Part G：质量控制与交付标准

## G.1 覆盖率自检矩阵

每份报告的第七篇§7.6必须包含16模块覆盖率自检表。

**V7增强**：相比V6的三列表格，V7增加"下一步"列，指导后续迭代方向。

| 模块 | 名称 | 覆盖状态 | 信息完整度 | 主要缺口 | 下一步 |
|------|------|---------|-----------|---------|--------|
| 1-16 | ... | 完成/部分/缺失 | 0-100% | 具体描述 | 搜索方向/采集计划 |

**完整度评估标准**：
- 95-100%：所有子项均有A/B级来源支撑
- 80-94%：核心子项完整，次要子项有C级来源
- 60-79%：框架存在但关键子项缺失
- 40-59%：仅有骨架性描述
- <40%：仅有标题和缺口标注

**合格线**：
- v1.0基线：平均≥40%
- v1.x数据补齐后：平均≥70%
- 最终版：平均≥75%，无模块低于50%

## G.2 交付物完整性检查

第八篇7项交付物的完整性标准：

| 交付物 | 合格标准 |
|--------|---------|
| §8.1 核心事实摘要 | 20-50条，每条含证据等级+具体来源URL |
| §8.2 政策立场矩阵 | ≥7个政策域，每个含立场/原话/证据/时期/一致性 |
| §8.3 盟友对手网络清单 | ≥8个组织/人物，含关系类型+性质+证据+战略意义 |
| §8.4 数据源索引 | 绑定到具体事实编号，非仅URL首页 |
| §8.5 信息需求清单 | ≥5项待采集缺口，含优先级和建议方向 |
| §8.6 整合时间轴 | 覆盖出生至今+前瞻2-3年，≥15个关键节点 |
| §8.7 争议矩阵 | ≥5事件，指控/证据/反证/结论四栏完整 |

## G.4 反幻觉检查清单

报告定稿前须逐项核实：

- [ ] 所有具体数字（得票数/金额/日期/百分比）均可溯源到A/B/C级来源
- [ ] 无"大约""据说""某些人认为"等模糊表述用于事实陈述（仅E级推断可用定性表述）
- [ ] 不同段落引用同一事实时，数字一致
- [ ] 中选会选举数据与官方公报一致
- [ ] 公督盟评鉴结果与官网记录一致
- [ ] 财产申报数据与监察院记录一致
- [ ] 无法溯源的数据均已标注【待核实】
- [ ] 不同LLM产出的同一事实已交叉验证，冲突处以A/B级来源为准

## G.7 新人物快速启动清单

为新的分析目标启动时，按以下清单执行：

1. [ ] 在 `data_collection/tasks/` 创建任务定义文件
2. [ ] 在 `_TASK_REGISTRY`（`run_collect.py`）注册新人物
3. [ ] 在 `perplexity_fetcher.py` 定义该人物的查询集（或复用通用查询）
4. [ ] 创建 `output/<人物名>/` 目录
5. [ ] 执行 Tier-0 采集：`--person <NAME> --tier 0`
6. [ ] 执行 LLM 提取
7. [ ] 获取台湾一手数据（Perplexity）
8. [ ] 生成 v1.0 基线报告
9. [ ] 运行覆盖率自检
10. [ ] 进入迭代循环（Phase 6）
